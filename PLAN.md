# PLAN — 첫 실전 학습 이후 로드맵 (2026-07-11)

첫 사전학습+SFT 완주 직후에 세운 계획. 원칙은 CLAUDE.md를 따른다:
새 기능은 "사람 마음의 어떤 측면을 흉내내는가"를 먼저 말하고, 공학적
함정을 솔직하게 평가한다. 예산 병목 없음(RunPod ~$11 ≈ 4090 25시간,
SFT 한 판 ≈ $0.5). 진짜 병목은 "뭘 할 가치가 있는가"다.

## 현황 스냅샷

- **base** `checkpoints/ckpt_best.pt` — full 프리셋(1B 토큰), val loss ~3.7대
- **sft.pt** — latent 2 + conf. 관찰: 단일턴 질답은 되나 "안녕~" 도배(디코딩
  루프), 멀티턴 무시(데이터에 없던 능력), 백과 헛소리 누출(base 누출).
  **긍정 신호**: 헛소리할 때 확신도가 0.7→0.52로 떨어짐 — conf 헤드가
  기능적으로 살아 있다는 첫 정황.
- 반복 페널티(rp=1.2 기본) 구현·검증 완료 — main 통합 대기.
- 파이프라인 스크립트로 무인 재개·Discord 알림·의존성 자가복구 동작 확인.

## Phase 1 — 멀티턴 (핵심, ~$1.5)

흉내내는 마음: 대화 문맥의 유지. "다른 말 해봐"가 통하는 최소 조건.
34M도 512토큰 창 안(한국어 4~6턴)의 얕은 문맥 참조는 학습 가능 —
지금 안 되는 건 능력 부재가 아니라 학습 신호 부재다.

- [ ] AI Hub 멀티턴 데이터(dataSetSn=71694) 다운로드 (계정+이용신청 필요)
- [ ] JSON 샘플 확인 → 변환기 작성 (`data/convert_aihub.py` 예정)
- [ ] `prepare_sft.py` 멀티턴 확장:
  - 템플릿 `<|user|>…<|end|><|assistant|>…<|end|>` 연쇄 (chat.py
    `build_prompt`와 이미 동형 — 추론 쪽 수정 불필요)
  - loss mask는 assistant 발화(+말미 `<|end|>`)만 1
  - 512 초과 대화는 최근 턴 우선으로 앞을 버림
- [ ] `make_latent_batch` 분기점을 첫 `<|assistant|>`에서 **마지막**으로 —
      "문맥을 다 읽고 → 잠재 사고 → 답변" 구조가 멀티턴의 의미에 맞다
- [ ] (페르소나, 거의 공짜) 변환/증강 단계에서 이름·말투 일관성 주입 —
      새 토큰·새 파라미터 없음, 데이터 수준 작업
- [ ] 멀티턴 SFT 조합: `--latent 2 --conf --workspace-slots 4 --mood-dim 64`
      → `sft_mt_ws_mood.pt`. **워크스페이스/기분이 처음으로 진짜 학습 신호를
      받는 순간** — 단일턴 데이터에선 "턴 사이 유지"가 배울 게 없었다.

## Phase 2 — 두 핵심 실험 (프로젝트의 존재 이유, ~$2)

- [ ] **latent vs pause**: 같은 base에서 `prepare_sft --n-pause 4` +
      `sft --n-pause 4` → `sft_pause4.pt` 학습 후
      `tools/eval_thinking --ckpts k0=… latent=… pause=…` —
      연속 잠재 사고가 이산 필러 속말보다 나은가 (같은 연산량 k에서)
- [ ] **캘리브레이션**: `tools/eval_conf --ckpt … --save` — 확신도 0.9
      구간에서 실제 ~90%를 맞히면 기능적 메타인지 성립. 위의 "헛소리 때
      확신 하락" 정황을 수치로 확정/기각
- [ ] 나머지 검증 D1~D6 (eval_state / eval_intervention / eval_integration /
      eval_schema) — 켠 기제가 있는 체크포인트별로, 전부 `--save`
- [ ] `BASELINE.md` 채우기 (사전학습 수치·SFT 조합별 masked val loss 표)

## Phase 4 — 자기개선: 대화 응고화 (₩0, Phase 1 이후)

흉내내는 마음: **수면 응고화** — 낮의 일화 기억(대화 로그)이 밤에
재생되며 가중치로 굳는다. rest drive가 높을 때 돌리면 "자면서 복습하는"
구조가 된다.

- [ ] chat.py에 대화 로그 저장 옵션
- [ ] `tools/consolidate.py`: 로그 → SFT 형식 → 현재 체크포인트에서
      저학습률 미세 SFT (노트북 CPU 밤새 가능, 34M라 현실적)
- 공학적 함정 (설계 제약으로 명시):
  1. **자기 출력 학습 금지** — 봇 답변을 타깃으로 쓰면 쓰레기 자기강화.
     기본은 사용자 발화만, 봇 답변은 명시적 승인분만.
  2. **파국적 망각** — 기존 SFT 데이터를 ~9:1로 재생(replay) 혼합.
  3. **기대치** — 정확한 개인 사실의 즉시 기억은 층위 1(state-file/
     workspace)의 일. 응고화는 말투·화제가 서서히 물드는 장기 기억이다.
- [ ] (선택) conf가 낮았던 구간을 응고화 우선순위로 — 메타인지가
      자기개선의 방향타를 잡는 배선

## 보류 결정 로그 (다시 제안되면 여기 참고)

- **웹서치/검색 도구**: 도구를 붙이는 건 쉬우나 34M이 검색 결과를 소화하지
  못한다(512 창 잠식 + 읽기 이해 부족). 도구와 도구를 쓸 지능은 별개.
- **사전학습 연장**: 코사인 스케줄이 감쇠를 끝내 이어붙이기가 지저분하고
  수확 체감(~3.7→3.5 추정). SFT 실험을 다 한 뒤 부족이 명확하면 재평가.
- 기존 보류 유지: FiLM 자기 억제, 레지스터 토큰. Block-Recurrent급
  read-write 메모리는 다음 세대 후보 그대로.
